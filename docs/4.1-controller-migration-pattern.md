## 4.1 Controller migration pattern – from legacy rights to permissions

This document corresponds to plan section **4.1** and todo
`controllers-pattern-doc`. It defines the standard approach for migrating
each controller from `checkRights('oldRightName')` and `Berechtigung`-based
logic to the new permissions/roles model.

### 1. Identify current check and target permission(s)

For each controller:

1. Locate existing checks:
   - `(new Helper\User($request, ...))->checkRights('oldRightName')`
   - `$workstation->checkRights('oldRightName')`
   - `$workstation->getUseraccount()->hasRights([...])`
2. Using the mapping docs and tickets (1172/1294/1295/1296), determine:
   - The **primary atomic permission** (e.g. `statistic`, `logs`,
     `appointment`, `organisation`, `dayoff`, `ticketprinter`, etc.).
   - Whether any additional permissions or role constraints are needed.

Record this in the controller’s migration checklist entry as:

- legacy right(s) → target permission(s) and/or “login only”.

### 2. Decide between permission vs. login-only

If the existing check is ambiguous (e.g. `checkRights()` without argument or
`basic`), classify the endpoint as one of:

- **Login only**:
  - Reads non-sensitive metadata or performs a harmless operation.
  - Replace `checkRights()` with a dedicated authentication guard and
    **no** permission requirement.
- **Permission required**:
  - Reads or writes sensitive data, or changes configuration/queues.
  - Replace `checkRights()` with a check that names the specific permission
    (e.g. `checkRights('availability')`, `checkRights('customersearch')`).

Document this choice in the checklist so that behaviour is explicit.

### 3. Apply the unified `hasRights()` semantics

Once the target permission(s) are known:

- Continue to use the existing `Helper\User::checkRights()` entry point, but:
  - Pass **permission names** instead of legacy rights whenever possible
    (e.g. `checkRights('statistic')` in Warehouse controllers).
  - Rely on `Useraccount::hasRights()` to:
    - Treat arguments that match permission keys as permission checks
      (via `permissions[...]`),
    - Fall back to legacy `rights.*` for old names during the migration
      period.

For multi-permission cases:

- Use `hasRights(['permA', 'permB'])` for all-of semantics.
- Where any-of semantics are required, call `hasAnyPermission()` directly on
  the `Useraccount` (or introduce a thin helper if necessary).

### 4. Keep legacy behaviour until the domain is fully migrated

During migration:

- Do **not** remove legacy rights checks immediately.
- Instead, for each controller:
  1. Switch the named check to the new permission (where applicable),
     keeping any additional legacy guards that are still needed.
  2. Add or update tests to assert the intended role/permission matrix.
  3. Only after the whole domain (e.g. statistics, 1294, 1295, 1296) is
     covered and tested, consider removing obsolete legacy-specific code
     paths as part of section 8.

### 5. Testing expectations per controller

For each migrated controller:

- Add tests that cover at least:
  - A role **with** the required permission(s) → access **allowed**.
  - A role **without** the required permission(s) → access **denied**.
  - Any documented “login only” case → authenticated user is sufficient.

For cross-domain tickets:

- 1172 (statistics vs. audit/logs):
  - `reporting_viewer` has `statistic`, not `logs`.
  - `audit_viewer` has `customersearch` + `logs`, not `statistic`.
- 1294 / 1295 / 1296:
  - Ensure queues, calendar, config, and emergency flows behave as described
    in the mapping tables.
