## 2.1 Backend rights API & helpers – inventory

This document corresponds to plan section **2.1** and todo
`inventory-backend-helpers`. It focuses on non-controller usages of:

- `checkRights()`
- `hasRights()`
- `testRights()`
- `isSuperUser()`

in helpers, entities, services, and collections.

### Helper `Zmsapi\Helper\User`

File: `zmsapi/src/Zmsapi/Helper/User.php`

- Static APIs:
  - `public static function checkRights(...)`
  - `public static function hasRights(...)`
  - `public static function testRights(...)`
  - `public static function hasXApiKey(...)`
- Behavioural usages:
  - Checks on `Useraccount`:
    - `hasRights(['department'])` (twice)
    - various `isSuperUser()` guard combinations
  - Delegates to `Workstation::getUseraccount()->testRights(...)`.

These methods are the primary backend entry points for controller-level
rights checking and will be migrated to the new permission model while
preserving their public signatures during the transition.

### Entity `Zmsentities\Useraccount`

File: `zmsentities/src/Zmsentities/Useraccount.php`

- Rights/permissions-related methods:
  - `public function hasRights(array $requiredRights)`
  - `public function testRights(array $requiredRights)`
  - `public function isSuperUser()`
- Internal usages:
  - `hasRights()` is used by:
    - `testRights()` for time-window based checks.
    - various domain entities/collections that look at high-level rights
      (`superuser`, `department`, etc.).

These methods are the core abstraction that must be extended to understand
atomic permission names while still supporting legacy right names.

### Entity helpers and collections

These classes call `Useraccount::hasRights()` to gate domain logic:

- `zmsentities/src/Zmsentities/Scope.php`
  - `hasRights(['superuser']) || $useraccount->hasScope($this->id)`.
- `zmsentities/src/Zmsentities/Department.php`
  - `hasRights(['superuser']) || $useraccount->hasDepartment($this->id)`.
- `zmsentities/src/Zmsentities/Owner.php`
  - `hasRights(['superuser'])` check for owner-related operations.
- `zmsentities/src/Zmsentities/Organisation.php`
  - `hasRights(['superuser'])` for organisation-level access.
- `zmsentities/src/Zmsentities/Collection/OrganisationList.php`
  - Iterates organisations, filtering based on `hasRights(['department'])`.
- `zmsentities/src/Zmsentities/Collection/UseraccountList.php`
  - Uses `hasRights($requiredRights)` for filtering.
  - Additional check for `hasRights(['department'])`.

These usages will need to be re-expressed in terms of explicit permissions
or roles where appropriate (e.g. `organisation`, `department`), once the
mapping is finalised.

### Service/helper classes

- `zmsapi/src/Zmsapi/Helper/ExchangeAccessFilter.php`
  - Several checks based on `static::$workstation->getUseraccount()->hasRights([...])`:
    - `['superuser']`
    - `['scope']`
    - `['department']`
    - `['organisation']`
  - These should map cleanly to `permissions.superuser`, `permissions.scope`,
    `permissions.department`, `permissions.organisation` once available.
- `zmsstatistic/src/Zmsstatistic/Helper/Access.php`
  - Uses:
    - `hasRights(['departmentStats'])` (legacy computed right).
    - `hasRights(['organisation'])`.
    - `isSuperUser()` for full access.
  - Will be migrated to `permissions.statistic` (and potentially
    organisation/department-based permissions) per ticket 1172.
